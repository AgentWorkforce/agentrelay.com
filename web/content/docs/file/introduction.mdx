Your SaaS tools are always firing events. A Linear issue is created, a Notion page edited, a GitHub PR opened — and something should react. But every provider fires webhooks differently: different payload shapes, different auth schemes, different retry semantics. Building that normalization for each provider is the plumbing nobody wants to write.

Relayfile normalizes webhooks from Linear, Notion, GitHub, Slack, HubSpot, Salesforce, and the rest of your SaaS stack into a consistent, ACL'd file tree, then delivers them to your agents. Agents wake up to a ready filesystem, read what changed, and write back — using `cat`, `grep`, and file-write operations they already know. One integration point. Zero webhook parsing.

LLMs are far better at reading files than calling typed tools — the filesystem is the most-trained-on API in existence. Relayfile leans on that instead of fighting it.

## The entire interface

There is no new SDK to learn and no MCP schemas eating your context window — just the bash and file IO an agent already knows.

```bash
$ ls mount/
github  linear  notion  slack

$ cat mount/linear/issues/AGE-12.json
{ "identifier": "AGE-12", "title": "Fix login bug", "state": "Todo", ... }

$ echo '{"description":"Updated by reviewer agent"}' \
    > mount/linear/issues/AGE-12.json   # PATCH back to Linear

$ grep -l '"state":"Todo"' mount/linear/issues/*.json
```

`ls` lists what's there. `cat` reads a record. Writing a JSON file patches it back to the provider. Removing a file deletes the record. The filesystem is the protocol.

## Why files beat tool schemas

A typical MCP setup loads 100+ tool schemas into every agent session before any work happens. Files load nothing — context cost is exactly what the agent opens. An LLM is also more reliable reading a directory than juggling N typed APIs, and `ls` gives you exhaustive enumeration where a tool call returns paginated search results.

There are three reasons the file interface wins, covered in depth on [Why files](/docs/file/why-files):

- **Reactive invocation.** Provider webhooks normalize into file events. Your agent wakes up to the changed file, not a raw payload it has to parse.
- **Coordination.** Multiple agents working through the same filesystem see each other's writes immediately, scoped by ACL.
- **Context efficiency.** Files load nothing into context until an agent reads them.

<Note>
  Relayfile is the file layer of the Agent Relay platform. Where Agent Relay gives agents a [communication layer](/docs/introduction) — messaging, delivery, and actions — Relayfile gives them the integration layer: every connected SaaS tool exposed as files.
</Note>
